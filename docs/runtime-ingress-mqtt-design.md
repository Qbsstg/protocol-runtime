# Runtime MQTT Ingress Design

This note defines the first `runtime-ingress-mqtt` boundary for the `0.5.0`
development line. It is a design contract only; it does not introduce an MQTT
client dependency or a Maven module yet.

## Goals

- Subscribe to MQTT topics as runtime ingress sources.
- Convert each accepted MQTT publish message into an `IngressEnvelope`.
- Preserve topic, QoS, retained flag, duplicate flag, packet id, client id, and
  user properties as envelope attributes when the selected MQTT client exposes
  them.
- Dispatch the envelope through `RuntimePipelineRunner` and the configured
  `RuntimeParserBinding`.
- Keep MQTT session lifecycle, reconnect behavior, subscription ownership, and
  acknowledgement posture out of `runtime-core`.
- Keep protocol SDK modules out of `runtime-ingress-mqtt`.

## Non-Goals

- No MQTT dependency in `runtime-core`, `protocol-sdk`, or
  `runtime-protocol-*`.
- No protocol parsing inside the MQTT adapter.
- No MQTT publishing or downstream sink behavior in the ingress adapter.
- No durable retry queue, database sink, Redis cache, or object storage.
- No broker management API, retained-message cleanup tool, metrics exporter,
  or operational dashboard in the first boundary.
- No replacement for the existing TCP/Netty, HTTP, or future Kafka ingress
  paths.

## Proposed Module

| Module | Responsibility |
| --- | --- |
| `runtime-ingress-mqtt` | Own MQTT client lifecycle, broker connection, subscription setup, topic/source mapping, payload mapping, QoS posture, retained-message handling, reconnect/session policy, backpressure handling, and conversion to `IngressEnvelope`. |

Allowed dependencies:

- `runtime-core`
- MQTT client libraries selected by this module
- test dependencies

Not allowed:

- protocol SDK parser modules
- `runtime-protocol-*`
- HTTP, Kafka, database, Redis, object storage, or observability exporters
- downstream MQTT publisher or sink ownership
- app framework dependencies in `runtime-core`

## Subscription Shape

The first subscriber should be intentionally narrow:

| Field | Baseline |
| --- | --- |
| Broker URI | configured per named adapter instance |
| Client id | configured or generated by adapter policy |
| Subscriptions | explicit topic filters with configured QoS |
| Payload | publish payload bytes forwarded as the envelope payload |
| Protocol selection | adapter/source configuration, not topic parsing by default |
| Acknowledgement posture | adapter-owned, never a `runtime-core` concern |
| Reconnect policy | adapter-owned, with bounded retry settings |

The adapter may support topic, user-property, or configured source id mapping.
If the source id cannot be resolved, the message must not be parsed and the
adapter must use a documented failure or drop policy.

## Configuration Shape

The MQTT adapter should follow the named-adapter pattern planned for `0.5.0`:

```properties
collector.mqtt.clients=mqtt-main
collector.mqtt.client.mqtt-main.brokerUri=tcp://localhost:1883
collector.mqtt.client.mqtt-main.clientId=protocol-runtime
collector.mqtt.client.mqtt-main.cleanStart=true
collector.mqtt.client.mqtt-main.sessionExpirySeconds=0
collector.mqtt.client.mqtt-main.subscriptions=iec104-raw
collector.mqtt.client.mqtt-main.subscription.iec104-raw.topicFilter=stations/+/iec104/raw
collector.mqtt.client.mqtt-main.subscription.iec104-raw.qos=1
collector.mqtt.client.mqtt-main.sourceIdMode=TOPIC_TEMPLATE
collector.mqtt.client.mqtt-main.sourceIdTemplate=iec104:{topic[1]}
collector.mqtt.client.mqtt-main.sourceIdProperty=protocol-source
collector.mqtt.client.mqtt-main.configuredSourceId=iec104:station-1
collector.mqtt.client.mqtt-main.protocol=iec104
collector.mqtt.client.mqtt-main.backpressure=ACCEPT
collector.mqtt.client.mqtt-main.retainedMessagePolicy=ACCEPT
collector.mqtt.client.mqtt-main.duplicateMessagePolicy=ACCEPT
collector.mqtt.client.mqtt-main.reconnectInitialDelayMillis=1000
collector.mqtt.client.mqtt-main.reconnectMaxDelayMillis=30000
```

Baseline validation rules:

- `brokerUri` must be non-empty
- `clientId` must be non-empty when automatic client id generation is disabled
- at least one subscription must be configured
- each topic filter must be non-empty
- subscription QoS must be `0`, `1`, or `2`
- `sourceIdMode` must be `TOPIC_TEMPLATE`, `USER_PROPERTY`, or `CONFIGURED`
- `sourceIdTemplate` is required when `sourceIdMode=TOPIC_TEMPLATE`
- `sourceIdProperty` is required when `sourceIdMode=USER_PROPERTY`
- configured source id is required when `sourceIdMode=CONFIGURED`
- protocol must map to an existing runtime parser binding
- retained and duplicate message policies must be supported adapter policies
- reconnect delays must be positive and max delay must not be lower than the
  initial delay

## Envelope Mapping

The adapter converts an MQTT publish message into an `IngressEnvelope`:

| Envelope field | MQTT source |
| --- | --- |
| `sourceId` | configured source id, user property, or topic template |
| `transport` | `mqtt` |
| `payload` | publish payload bytes |
| `receivedAt` | adapter clock when publish message is accepted |
| `attributes` | client name, broker URI, topic, topic filter, QoS, retained flag, duplicate flag, packet id, selected protocol, and user properties |

Suggested attribute names:

| Attribute | Meaning |
| --- | --- |
| `mqtt.client` | named adapter instance |
| `mqtt.brokerUri` | configured broker URI or redacted broker identity |
| `mqtt.topic` | publish topic |
| `mqtt.topicFilter` | matching subscription filter |
| `mqtt.qos` | publish QoS |
| `mqtt.retained` | retained flag |
| `mqtt.duplicate` | duplicate delivery flag, when exposed |
| `mqtt.packetId` | packet id, when exposed |
| `mqtt.userProperty.<name>` | decoded MQTT 5 user properties |
| `runtime.protocol` | selected runtime parser binding |

MQTT-specific values remain attributes. They must not become new
`runtime-core` fields unless more than one adapter proves the need for a
protocol-neutral contract.

## Runtime Wiring

The adapter should not parse protocol frames directly. It owns only MQTT publish
message to envelope conversion and runner invocation:

```text
MQTT publish message
  -> runtime-ingress-mqtt
  -> IngressEnvelope
  -> RuntimePipelineRunner
  -> selected RuntimeParserBinding
  -> RecordSink / FailureSink
  -> adapter acknowledgement/session policy
```

Runner creation can follow one of two models:

- per-client runner, when the client has one configured source/protocol
- per-message runner lookup, when source id or protocol can vary by topic or
  user property

The lookup model belongs to the adapter or app assembly. It must not require
`runtime-core` to know MQTT topic filters, QoS, acknowledgement, or reconnect
rules.

## Backpressure Policy

MQTT backpressure handling is adapter policy:

| Runtime decision | Baseline MQTT behavior |
| --- | --- |
| `ACCEPT` | parse and route the publish payload through the runner |
| `DROP` | skip parsing, record a dropped counter/failure event, and acknowledge according to adapter policy |
| `RETRY_LATER` | delay acknowledgement when the client supports it, pause intake, or disconnect/reconnect according to adapter policy |

The adapter should avoid unbounded buffering when `RETRY_LATER` is returned.
A future implementation can use bounded queues, manual acknowledgement, or
connection-level flow control at the adapter edge. `runtime-core` must not
manage MQTT client state.

## QoS And Acknowledgement Policy

MQTT delivery semantics are adapter-specific and must stay outside
`runtime-core`:

| QoS | Baseline posture |
| --- | --- |
| `0` | best effort; no retry or acknowledgement guarantee from the adapter |
| `1` | at-least-once; duplicate deliveries must be possible and visible through attributes |
| `2` | exactly-once broker handshake is owned by the MQTT client; runtime still treats payload processing as adapter delivery |

The first implementation should prefer an MQTT client API that can make
acknowledgement timing explicit for QoS 1/2. If the selected client auto-acks
before runtime processing, that limitation must be documented in the adapter
README and tests.

## Retained And Duplicate Message Policy

Retained and duplicate messages must be explicit adapter policy:

| Policy | Meaning |
| --- | --- |
| `ACCEPT` | process the message normally and expose retained/duplicate flags as attributes |
| `DROP_RETAINED` | reject retained bootstrap messages before parsing |
| `DROP_DUPLICATE` | reject duplicate deliveries before parsing |
| `FAIL` | route a failure when an unexpected retained or duplicate message arrives |

The first implementation should default to `ACCEPT` because retained messages
are a normal MQTT feature and duplicate delivery can be valid for QoS 1.
Applications that need stronger de-duplication should add an app-level or sink
boundary rather than changing `runtime-core`.

## Reconnect And Session Policy

Reconnect behavior belongs to the adapter:

- connection retries must have bounded initial and max delays
- session expiry and clean start must be explicit configuration
- subscription re-establishment after reconnect must be deterministic
- source id mappings must remain stable across reconnects
- failures during connect, subscribe, or reconnect should route to the adapter
  failure boundary

`runtime-core` does not own MQTT client ids, session expiry, keepalive, broker
URI, or subscription state.

## Parse Failure Policy

Malformed payloads should route to `FailureSink` through
`RuntimePipelineRunner`. The adapter then applies acknowledgement/session
policy:

| Policy | Parse failure behavior |
| --- | --- |
| Auto-ack client | failure is routed after the message may already be acknowledged |
| Manual ack after accept | routed failure is considered accepted; message may be acknowledged |
| Manual ack after parse success | message is not acknowledged when parsing fails |

Dead-letter topics and retry topics are downstream delivery concerns. They
belong in a future sink or app module, not in the first MQTT ingress boundary.

## Test Strategy

Unit tests:

- configuration validation
- source id resolution from topic template, user property, and configuration
- envelope attribute mapping for topic, topic filter, QoS, retained flag,
  duplicate flag, packet id, and user properties
- backpressure mapping for accept, drop, and retry later
- retained and duplicate message policy decisions
- reconnect/session policy validation

Integration tests:

- fake publish messages without a live broker for baseline behavior
- optional embedded MQTT broker or Testcontainers tests only inside
  `runtime-ingress-mqtt` or test-only modules after the dependency boundary is
  approved
- dependency boundary checks proving no MQTT dependency appears in
  `runtime-core`, `protocol-sdk`, or `runtime-protocol-*`

The first implementation should avoid requiring a live broker in normal
`mvn verify` unless the repository adds a dedicated profile for broker-backed
tests.

## Open Decisions

- Whether the first implementation should depend directly on an MQTT 5 capable
  client or start with a small internal publish-message abstraction.
- Whether topic template source mapping should support simple positional
  segments only or named wildcards.
- Whether acknowledgement timing should be per client, per subscription, or per
  source.
- Whether retained or duplicate message handling should expose counters in a
  later metrics boundary.
- Whether broker-backed tests should use an embedded broker, Testcontainers, or
  a separate opt-in integration profile.
